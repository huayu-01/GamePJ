# 热更新与版本发布流程

## 更新边界

- APK/AAB 更新：C# 对局规则、网络协议、原生功能和安全修复。
- PCK 热更新：图片、音频、文本、主题和不包含 C# 的资源场景。
- 远程配置：公告、维护开关以及服务端认可的数值配置。

核心发牌、行动、分池和结算逻辑不得通过下载代码动态替换。

## 配置更新服务器

项目默认从 GitHub Raw 获取正式清单：

```ini
[game]

update/manifest_url="https://raw.githubusercontent.com/huayu-01/GamePJ/main/update/manifest.json"
```

正式清单位于 `update/manifest.json`，模板位于 `update/manifest.example.json`。开发环境也可以设置 `GAMEPJ_UPDATE_MANIFEST_URL` 覆盖仓库地址。正式包只接受 HTTPS；调试包额外允许本机 HTTP 地址。

GitHub Raw 只用于体积很小的 JSON 清单。APK 可以使用仓库中的 Git LFS 下载地址；PCK 和正式 APK 更推荐上传为 GitHub Release Asset，再把下载地址写入清单。私有仓库不能把访问令牌打包进客户端，因此本方案要求更新文件公开可读。

## 清单规则

- `latest_app_version`：最新客户端版本。
- `minimum_app_version`：低于此版本时提示强制更新。
- `protocol_version`：必须与 `Constants.NetworkProtocolVersion` 相同。
- `apk_url`：新 APK 或发布页地址。
- `config`：保存至本地缓存，可通过 `UpdateManager.GetRemoteString/GetRemoteInt/GetRemoteBool` 读取。
- `packs`：PCK 资源包列表；每个包必须提供 HTTPS 地址和 SHA-256。

客户端先下载到 `.download` 临时文件，核对长度与 SHA-256 后才原子替换。旧包保留为 `.previous`。新包在下次启动时加载；加载失败会自动恢复上一版。

## 制作 PCK

资源包不能包含 C# 代码。通过 Godot 的 PCK/ZIP 导出功能生成资源包，计算 SHA-256：

```powershell
Get-FileHash .\base-art.pck -Algorithm SHA256
```

上传 PCK 后更新清单中的 `version`、`url`、`sha256` 和 `size`。先用测试清单验证，再替换正式清单。

## GitHub 发布顺序

1. 导出并测试 APK/PCK。
2. 上传 GitHub Release Asset 或 Git LFS 文件。
3. 取得最终下载地址并计算 PCK 的 SHA-256 与字节大小。
4. 最后更新 `update/manifest.json` 并推送。清单必须最后发布，避免客户端看到尚未上传完成的资源。
5. 仅更新资源时保持客户端版本不变，只更新对应 PCK 的 `version`、`url`、`sha256` 和 `size`。
6. 更新 C# 或网络协议时提升 `latest_app_version`、Android `version/code`；协议不兼容时同时提升 `protocol_version` 和 `minimum_app_version`。

APK 文件名必须包含 `version/name`，统一为 `GamePJ-<版本号>.apk`。使用 `tools/android/export_versioned_apk.ps1` 导出，避免清单、Git LFS 文件和实际版本错位。

## 网络版本

公网房间目录和玩家注册都必须携带协议版本。不同协议版本的房间不会出现在自动发现结果中；直接连接调试时，服务器会拒绝不兼容客户端并显示更新提示。

## 回滚

1. 资源问题：在远程清单中恢复上一份 PCK 的地址和 SHA-256，客户端会重新下载。
2. 本地包损坏：启动加载失败时自动恢复 `.previous`。
3. 客户端问题：发布更高 `version/code` 的 APK，不要复用已经发布的 Android 版本号。
